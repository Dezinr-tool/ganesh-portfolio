# Google Ventures Design Sprint

## Overview

The Design Sprint is a five-day process for answering critical business questions through design, prototyping, and testing with real users. Created by Jake Knapp at Google Ventures (now GV), it compresses months of debate into a structured week that produces a tested prototype and clear next steps.

The sprint is not a substitute for full product development. It is a **decision-acceleration and risk-reduction tool** that validates or invalidates hypotheses before significant engineering investment.

## The Five-Day Schedule

### Monday — Map and Choose

**Theme:** Understand the problem and pick a target.

| Time | Activity |
|------|----------|
| 10:00 | Start at the End — Define long-term goal and sprint questions |
| 11:00 | Map — Draw user journey; identify actors, steps, and pain points |
| 12:00 | Lunch |
| 13:00 | Ask the Experts — Lightning talks from stakeholders (15 min each) |
| 15:00 | Target — Select one critical moment in the journey to focus the sprint |
| 16:00 | Note and Vote — HMW notes, cluster, dot-vote on most important area |

**Outputs:** Sprint goal, journey map, target moment, list of HMW opportunities.

**Facilitator focus:** Prevent scope creep. One target moment, not the entire product.

### Tuesday — Sketch Solutions

**Theme:** Generate solutions individually, then critique collectively.

| Time | Activity |
|------|----------|
| 10:00 | Lightning Demos — Review inspiring solutions from other domains (3 min each) |
| 11:00 | Four-Step Sketch — Notes → Ideas → Crazy 8s → Solution Sketch |
| 12:00 | Lunch |
| 13:00 | Continue Solution Sketches (anonymous, detailed, self-explanatory) |
| 15:00 | Art Museum — Silent gallery walk of all sketches |
| 15:30 | Heat Map — Dot stickers on compelling ideas (silent) |
| 16:00 | Speed Critique — 3 min per sketch, presenter doesn't speak |
| 16:45 | Straw Poll — Each person votes for one concept |
| 17:00 | Supervote — Decider picks 1–3 concepts to prototype |

**Outputs:** 1–3 solution sketches selected for prototyping.

**Facilitator focus:** Enforce anonymity in sketch phase to reduce hierarchy bias. Decider must participate, not delegate.

### Wednesday — Decide and Storyboard

**Theme:** Resolve ambiguity and plan the prototype.

| Time | Activity |
|------|----------|
| 10:00 | Critique and compare winning sketches |
| 11:00 | Rumble (if 2–3 concepts) or direct selection |
| 12:00 | Storyboard — 10–15 panel comic of user flow through prototype |
| 13:00 | Lunch |
| 14:00 | Continue storyboard; assign prototype roles |
| 16:00 | Review storyboard with Decider for final approval |

**Outputs:** Detailed storyboard, prototype assignment (Maker, Stitcher, Writer, Interviewer).

**Facilitator focus:** Storyboard must be testable in 15 minutes of user time. Cut nonessential screens.

### Thursday — Prototype

**Theme:** Build a realistic facade, not a functional product.

| Time | Activity |
|------|----------|
| All day | Build prototype using chosen fidelity tool |
| 16:00 | Trial run — Team walks through prototype; fix gaps |
| 17:00 | Write interview script and screener criteria |

**Outputs:** Clickable prototype (Figma, Keynote, or HTML), interview script.

**Rules:**
- Pick one tool and commit; no tool debates on Thursday.
- Realistic data and copy; no lorem ipsum.
- Prototype only the test path, not edge cases.

### Friday — Test

**Theme:** Learn from five 1-on-1 interviews.

| Time | Activity |
|------|----------|
| 9:00–12:00 | Five 40-minute user interviews (prototype + debrief) |
| 12:00 | Lunch |
| 13:00 | Pattern synthesis — Green/yellow/red sticky notes on observations |
| 15:00 | Sprint retrospective and next-step decision |

**Outputs:** Validated or invalidated hypothesis, patterns across interviews, recommended next steps.

**Facilitator focus:** Interviewer asks open questions; team watches silently (live or recorded). No defending the prototype.

## Roles

| Role | Responsibility |
|------|----------------|
| **Facilitator** | Runs schedule, enforces timeboxes, manages energy |
| **Decider** | CEO, PM, or person with final call — breaks ties |
| **Maker** | Builds prototype screens |
| **Stitcher** | Connects screens, handles transitions |
| **Writer** | Realistic copy, error messages, microcopy |
| **Interviewer** | Conducts Friday tests; neutral, curious tone |
| **Experts** | Monday lightning talks; available for questions |

Minimum team: 4 people. Ideal: 5–7. Decider attendance on Monday, Wednesday, and Friday is non-negotiable.

## When to Use a Design Sprint

**Good fit:**
- High-stakes decision with stakeholder disagreement
- New product or major feature with unclear direction
- Team stuck in analysis paralysis for 2+ weeks
- Need to test risky assumption before Q planning commitment
- Post-research synthesis needs rapid concept validation

**Poor fit:**
- Problem already validated and solution scoped (just build it)
- No access to target users within 1 week
- Decider unavailable for key decision moments
- Technical spike is the primary unknown (run engineering spike instead)
- Incremental optimization (A/B test instead)

## Facilitation Best Practices

1. **Timebox ruthlessly** — Use a visible timer. Move on when time expires.
2. **Work alone together** — Individual sketching before group discussion reduces groupthink.
3. **Silent voting first** — Heat maps and dot votes happen before discussion.
4. **Capture HMWs continuously** — Monday expert talks generate opportunities; don't lose them.
5. **Protect Friday interviews** — Cancel internal meetings. Testing is the sprint's ROI.
6. **Document as you go** — Photos of sketches, storyboard, and Friday board become the deliverable.
7. **Manage the Decider** — Brief them pre-sprint on their role: decide, don't defer.

## Remote Adaptations

Remote sprints require intentional tooling and rhythm changes.

### Tooling Stack
- **Video:** Zoom or Google Meet with breakout rooms
- **Whiteboard:** FigJam or Miro (persistent boards per day)
- **Sketching:** Figma (individual pages per person) or paper + phone photo upload
- **Prototyping:** Figma (shared file, component library pre-loaded)
- **Testing:** UserTesting, Maze, or Zoom screen-share with Lookback
- **Async:** Loom for lightning demos if timezone spread

### Schedule Adjustments
- **Shorten days to 6 hours** — Video fatigue reduces productive hours.
- **Split Tuesday sketch across Mon evening + Tue** — Four-Step Sketch benefits from overnight incubation.
- **Pre-sprint async prep** — Share expert interview recordings before Monday.
- **Dedicated Slack channel** — Async questions only; no side debates in DMs.
- **Friday across timezones** — Book users in Decider's timezone; record for async team review.

### Remote-Specific Risks
- Heat map voting is harder — use FigJam voting plugin or numbered stickers.
- Silent critique breaks down with eager talkers — Facilitator mutes all except presenter proxy.
- Prototype handoff — Assign one Maker with Figma expertise; avoid multi-tool confusion.

## Variations

### 4-Day Sprint (Drop Monday Mapping)

Use when problem is pre-mapped from prior research.
- **Day 1:** Expert review + sketch (compressed Tuesday)
- **Day 2:** Decide + storyboard
- **Day 3:** Prototype
- **Day 4:** Test

Requires pre-sprint deliverable: journey map, target moment, sprint questions.

### 3-Day Sprint (Concept Sprint)

Use for lower-stakes direction setting without user testing.
- **Day 1:** Map + sketch
- **Day 2:** Decide + storyboard + prototype
- **Day 3:** Internal critique + stakeholder review (no user tests)

Trade-off: Faster but no user evidence. Follow with testing within 2 weeks.

### Design Sprint 2.0 Modifications (Knapp updates)

- **Lightning talks can happen async** before sprint week.
- **"Note and Vote" on Map** — Team suggests target moments; Decider chooses.
- **AI tools** — Use for demo research and copy drafts on Thursday, not for skipping user tests.

## Common Failures

| Failure | Symptom | Prevention |
|---------|---------|------------|
| No Decider | Endless debate, no convergence | Confirm Decider commitment pre-sprint |
| Wrong problem | Friday tests confuse users | Invest in Monday mapping; reject pre-written solutions |
| Over-scoped prototype | Thursday crunch, incomplete flow | Storyboard max 15 panels |
| Fake Friday | Internal team "pretests" instead of real users | Recruit users 2 weeks ahead; offer incentive |
| Solution attachment | Team ignores negative test results | Facilitator frames Friday as learning, not validation theater |
| Skipping sketch anonymity | HiPPO picks their idea early | Enforce silent gallery and heat map |
| Engineering absent | Prototype ignores feasibility | Include tech lead in Monday experts |
| No follow-through | Sprint deck shelved | End Friday with explicit next-step owner and date |

## Post-Sprint Decision Framework

After Friday synthesis, choose one:

1. **Go** — Proceed to build; create backlog from storyboard.
2. **Iterate** — Run another sprint on refined target (common after partial validation).
3. **Pivot** — Change target moment or problem based on learnings.
4. **Stop** — Kill the initiative; document why (successful outcome if it saves months).

## Templates

### Sprint Brief

```markdown
# Design Sprint Brief

## Challenge
[One sentence]

## Decider
[Name]

## Sprint Team
| Name | Role | Availability Mon–Fri |

## Sprint Questions (max 3)
1. ...
2. ...
3. ...

## Long-Term Goal
[6–12 month success state]

## Target Users for Friday
[Segment, recruitment plan, incentive]

## Pre-Work Completed
- [ ] Journey map draft
- [ ] Expert interviews scheduled
- [ ] Users recruited (5 confirmed)
```

### Friday Interview Script Outline

```markdown
1. Welcome and context (2 min)
2. Background questions — non-leading (5 min)
3. Task scenario introduction (1 min)
4. Prototype tasks — think aloud (15 min)
5. Debrief questions (10 min)
6. Thank you and incentive (2 min)

Rules: Don't help. Don't explain the UI. Ask "what are you thinking?"
```
