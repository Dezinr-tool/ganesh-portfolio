# How Might We (HMW)

## Overview

"How Might We" (HMW) is a question-framing technique used in design thinking to convert research insights and problem statements into actionable opportunity areas for ideation. The format was popularized by IDEO and the Stanford d.school as a bridge between the Define and Develop phases of the design process.

An HMW question opens solution space without prescribing an answer. It is intentionally optimistic ("might"), collaborative ("we"), and exploratory ("how").

## The HMW Format

```
How might we [desired outcome]
for [user/context]
without [constraint to avoid]
```

Not all parts are required. The minimum viable HMW:

```
How might we [verb + outcome area]?
```

### Anatomy of a Good HMW

| Component | Purpose | Example |
|-----------|---------|---------|
| **How** | Invites exploration, not certainty | "How" not "We should" |
| **Might** | Allows multiple solutions; no commitment | "Might" not "Will" |
| **We** | Implies team ownership and collaboration | "We" not "The product" |
| **Action area** | Focuses on outcome or behavior change | "...help users feel confident..." |
| **Optional: context** | Grounds in situation | "...during first-time setup..." |
| **Optional: constraint** | Prevents obvious but wrong solutions | "...without adding more steps..." |

## IDEO Process Context

HMW questions sit at a specific moment in the IDEO/design thinking flow:

```
Empathize → Define → IDEATE → Prototype → Test
              ↓           ↑
         Point of View   HMW questions
         + Insight       fuel brainstorm
```

### From POV to HMW

**Point of View (POV) statement:**
```
[User] needs [need] because [insight].
```

**Convert to HMW:**
Take the need and reframe as an open question.

POV:
> A first-time freelancer needs to understand tax obligations because unexpected bills create financial anxiety and distrust of the platform.

HMW:
> How might we help first-time freelancers anticipate tax obligations before they receive a surprise bill?

### Multiple HMWs from One Insight

One insight should generate **3–5 HMW questions** at different altitudes:

Insight: Users abandon onboarding at the bank connection step.

HMW variations:
1. **Narrow:** How might we explain why bank connection is needed before asking for credentials?
2. **Medium:** How might we build trust during financial data collection?
3. **Broad:** How might we reduce anxiety during onboarding?
4. **Constraint:** How might we verify identity without connecting a bank account?
5. **Analogous:** How might we make financial setup feel as easy as signing up for email?

Generate breadth before selecting focus.

## Workshop Facilitation

### HMW Generation Session (60–90 minutes)

**Participants:** 4–8 cross-functional members (design, PM, engineering, support, sales)

**Materials:** Sticky notes, whiteboard/FigJam, research synthesis visible

#### Step 1: Share Insights (15 min)
Facilitator presents 3–5 top research insights with evidence (quotes, clips, data). No solutions yet.

#### Step 2: Silent HMW Writing (10 min)
Each participant writes as many HMW questions as possible—one per sticky note. Aim for 5–10 each.

Rules:
- Start every note with "How might we..."
- No self-censoring
- Build on insights, not assumptions

#### Step 3: Share and Cluster (15 min)
Read HMWs aloud (no debate). Cluster similar questions on the board. Merge duplicates.

#### Step 4: Vote (5 min)
Each person gets 3 dot votes. Top 3–5 HMWs become ideation prompts.

#### Step 5: Critique HMW Quality (10 min)
Review winners against good/bad criteria (see below). Rewrite weak HMWs before ideation.

#### Step 6: Ideation (30 min)
Run Crazy 8s or brainwriting on the top HMW. One HMW at a time.

### Facilitation Tips

- **Enforce "How might we" prefix** — Prevents solution statements sneaking in.
- **Ban "How might we build X"** — "Build a chatbot" is a solution. "Help users get answers at 2am" is an HMW.
- **Timebox silent writing** — Group discussion before individual generation kills diversity.
- **Keep research visible** — HMWs drift toward assumptions when evidence isn't present.
- **One ideation session per HMW** — Don't combine multiple HMWs in one brainstorm; scope explodes.

## Good vs Bad HMW Questions

### Good HMW Examples

```
How might we help patients remember pre-appointment preparation steps?
```
✅ Outcome-focused, user-centered, open to many solutions (SMS, email, app, call)

```
How might we reduce the time it takes for new engineers to ship their first PR?
```
✅ Measurable direction, specific audience, no prescribed solution

```
How might we make refund status transparent without increasing support call volume?
```
✅ Includes constraint that prevents naive solutions

```
How might we celebrate user progress to reinforce daily habits?
```
✅ Emotional outcome, open solution space

### Bad HMW Examples

```
How might we build a dashboard?
```
❌ Solution disguised as question. Better: "How might we help managers understand team performance at a glance?"

```
How might we increase revenue?
```
❌ Too broad, business-centric not user-centric. Better: "How might we help users discover premium features they would find valuable?"

```
How might we fix the login bug?
```
❌ Not a design opportunity; it's a ticket. Better: "How might we help returning users access their account effortlessly?"

```
How might we design a better app?
```
❌ Meaningless without specificity. Better: anchor in insight.

```
How might we use AI to personalize content?
```
❌ Technology-first. Better: "How might we surface content relevant to each user's current project?"

### HMW Quality Rubric

| Criterion | Good | Bad |
|-----------|------|-----|
| **Scope** | Focused enough for 30-min brainstorm | Too broad (entire product) or too narrow (one button) |
| **User-centered** | Describes user outcome | Describes business metric or feature |
| **Solution-free** | Multiple solutions possible | Only one obvious answer |
| **Evidence-based** | Traces to research insight | Based on stakeholder opinion |
| **Optimistic** | Possibility framing | Problem restatement ("HMW fix broken...") |
| **Actionable** | Team can ideate immediately | Requires more research first |

## Altitude Framework

HMW questions operate at different zoom levels:

```
Broad  ────────────────────────────────────  Narrow
"Improve health"    "Improve medication adherence"    "Help elderly users read pill labels"
```

**Workshop strategy:**
1. Start broad to explore solution space.
2. Narrow for sprint focus.
3. Document both—broad HMWs become roadmap themes; narrow HMWs become sprint prompts.

## Remote HMW Workshops

- **FigJam/Miro template:** Columns for Insight → HMW → Cluster → Vote → Ideate
- **Async HMW generation:** Participants add stickies before live session; cluster live
- **Dot voting:** Use built-in voting or emoji reactions
- **Breakout rooms:** Small groups each tackle one HMW for ideation
- **Record insight presentation:** Loom video for timezone-async teams

## Combining HMW with Other Frameworks

| Framework | HMW Role |
|-----------|----------|
| **Double Diamond** | Define → Develop bridge |
| **Design Sprint** | Monday "Note and Vote" on HMWs from expert talks |
| **JTBD** | Reframe job motivations as HMW questions |
| **Jobs Stories** | Convert "I want to..." into "How might we help users..." |
| **Lean UX** | HMWs become hypothesis starting points |
| **Service Design** | Generate HMWs per journey phase or pain point |

## Common Mistakes

1. **One HMW only** — Teams pick the first question without exploring alternatives. Generate many, select deliberately.

2. **HMW as deliverable** — Writing HMWs without following ideation. The question exists to spark ideas, not to fill a deck.

3. **Stakeholder HMWs without research** — Executives generate HMWs from assumptions. Ground in empathy work first.

4. **Negative framing** — "How might we stop users from churning" focuses on problem. Reframe: "How might we help users achieve their first value moment within 24 hours?"

5. **Premature convergence** — Voting on HMWs before generating breadth. Always generate 20+ before selecting 3.

6. **Ignoring constraints** — Without "without adding steps" or similar, teams default to lazy solutions (more notifications, more modals).

## Case Study: IDEO Shopping Cart Project

IDEO's famous shopping cart redesign for ABC News began with field research in supermarkets. An insight: parents struggle to manage kids while shopping and checking out.

HMW generated:
- How might we keep children engaged during shopping?
- How might we simplify checkout for distracted parents?
- How might we reduce the physical awkwardness of the cart?

These HMWs led to divergent concepts—not "better cart wheels" but RFID scanners, child seats facing parents, and modular basket designs. The HMW framing prevented the team from fixating on incremental cart improvements.

## Templates

### HMW Workshop Board (FigJam/Miro)

```
| INSIGHTS (evidence) | HMW QUESTIONS | CLUSTERS | TOP 3 (voted) | IDEAS |
```

### HMW Worksheet

```markdown
## Research Insight
[Quote, observation, or data point]

## POV Statement
[User] needs [need] because [insight].

## HMW Variations
1. Narrow: How might we ...
2. Medium: How might we ...
3. Broad: How might we ...
4. With constraint: How might we ... without ...
5. Analogous: How might we ... (inspired by [other domain])

## Selected HMW for Ideation
[Which one and why]

## Success Criteria
[How we'll know ideation addressed the right problem]
```
